线程的实现方式

  • 继承Thread类，重写run方法，通过start方法启动
  • 实现Runnable接口，重写run方法（创建Thread对象，利用Thread对象来启动线程）
  • 实现Callable接口，重写call方法

线程的定义

  • 顶级父类Thread
class MyThread extends Thread{ // 继承Thread
    public void run(){ //  重写run方法
    
    }
}

// 获取线程对象
this.getName();
// 设置线程对象的名字
Thread t1 = new Thread();
t1.setName();
  • Runnable接口
class MyRunnable implements Runnable{ // 实现Runnable接口
    @Override
    public void run(){ // 重写run方法
    }
   
}
// 创建自定义类对象
MyRunnable mr= new MyRunnable(); 
Thread t = new Thread(mr);// 将其组为参数传递给Thread的构造函数
t.start(); // 开启线程

// 获取当前线程的对象
Thread.currentThread().getName();
// 设置线程对象的名字
Thread.currentThread().setName();

区别

  • 没有start()方法，只能通过Thread开启线程
  • 一个Thread对应一个线程，不能多次开启
  • new Thread(); 可以多个相互独立

线程同步

例如：银行取钱
银行存款有一百万【共同资源】，现在有十个人【多条线程】 同时来取钱，每个人都想取十万。每个人都直接修改这个【共同资源】的数据，一瞬间全部完全取款。结果银行存款【共同资源】剩下100-10=90万（因为瞬间还没等别人处理完，都拿到这个资源，并且去修改，所以每个人拿到的原始值都是100，算出的结果都是剩下90），每个人也都拿到钱，凭空生出了九十万，这明显是不对的，这是不安全的。

为了保证共同资源的安全，应该让他们一个一个的去使用这个资源，第一个取完剩下九十万，第二个取完剩下八十万…最终全部取万剩下零元。

线程锁

为了保证共享资源的安全，把资源加锁，有人在用的时候锁死，不用的时候开锁

  • Synchronized

    • synchronized不需要用户去手动释放锁，当synchronized方法或者synchronized代码块执行完之后，系统会自动让线程释放对锁的占用

  • Lock

    • Lock则必须要用户去手动释放锁，如果没有主动释放锁，就有可能导致出现死锁现象

  • ReentrantLock类 (可重入锁)

  • ReadWriteLock接口 (排它锁)

    • 具有完全互斥排他的效果，即同一时刻只允许一个线程访问

  • ReentrantReadWriteLock

    • ReentrantReadWriteLock实现了上面讲的ReadWriteLock接口
